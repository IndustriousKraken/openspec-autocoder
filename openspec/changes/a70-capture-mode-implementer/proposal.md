## Why

The implementer is the one role still pinned to the `claude` strategy. `a60` left the streaming implementer on `claude`, and `a56` keeps `final_answer` / `session_id` / the live log claude-specific. But the implementer's *structured outcome already rides the MCP `outcome_*` / `record_outcome` relay*, so the implementer can run through a capture-mode strategy with only the live incremental log lost. Generalizing it unblocks `opencode` (which matches `claude` in capability) AND `gemini` as operator-selectable implementers — and through `opencode`, the broader set of provider-routed models. This is opt-in; some projects are fine with a Gemini or OpenCode implementer, and for an operator who only has a Gemini subscription it beats a local Ollama model.

AskUser is handled by each CLI's **native headless resume** (`claude` via the captured `session_id`, `opencode` via `--session`, `gemini` via `--resume`). A resume that cannot be restored is treated as a broken attempt and the change is **requeued** — there is no stash-and-recombine fallback. A fallback that only fires when the session store is wiped is an untested branch that hides a real failure; requeuing reuses the existing failure-counter path and escalates to perma-stuck if it keeps failing, which is the signal an operator wants.

Because the daemon spawns a session per change, it must own session hygiene: Claude's `~/.claude` is known to grow unbounded (anthropics/claude-code #24207, with reports of 200–472 GB and cascade failures that destroy settings and auth at 0 bytes free). The implementer therefore prunes the specific session it created on completion — **surgically**.

## What Changes

**The implementer runs through any `CliStrategy`, not `claude` alone.** For a capture-mode strategy (`opencode`, `gemini`), the implementer runs via `agentic_run` in capture mode, with the structured outcome AND the agent's `final_answer` delivered through the MCP outcome relay rather than parsed from streaming-JSON. The streaming (live-log) implementer path remains claude-specific (per `a60`); a capture-mode implementer simply runs without the live log. The default implementer is unchanged (`claude`).

**Every agentic role cleans up the session it creates.** The CLIs persist a transcript per invocation in the home directory, so left alone they accumulate without bound. Every role that runs through `agentic_run` — the implementer AND the single-shot roles (advisory audits, reviewer, contradiction check) — removes the session it created when it is done with it: single-shot roles on completion, the implementer on its terminal outcome. The prune is **surgical** — only the specific session record, by its identifier, via the CLI's own session-delete — never settings, memory/context files (`CLAUDE.md` / `GEMINI.md` / project memories), credentials, or the generated MCP config. The principle: a run that drops litter — even when it is upstream software writing into our home directory — cleans it up at the end.

**The implementer also handles AskUser via native resume.** On AskUser it submits the question via the outcome relay, ends the run with the change waiting, AND retains its session; on the operator's answer it resumes that same session via the strategy's native headless mechanism (`session_id` / `--session` / `--resume`). If the session cannot be restored (not found / corrupt / expired by the CLI's own retention), it does NOT fall back to a fresh run — it requeues the change as a retryable failure (no stash-and-recombine).

## Impact

- **Affected specs:** `executor` — ADD `Implementer runs through any CliStrategy (capture-mode path)`; ADD `Every agentic role cleans up the session it creates`; ADD `Implementer resumes its session on AskUser; resume failure requeues`.
- **Affected code:** the implementer dispatch (resolve strategy from the model, like other roles); the `CliStrategy` trait gains a headless-resume mechanism AND a scoped session-delete; capture-mode `final_answer` carried via the outcome submission; resume-failure → requeue via the existing failure-counter; surgical session prune on terminal outcome.
- **Operator-visible behavior:** `opencode` and `gemini` become selectable implementers (opt-in; default stays `claude`); capture-mode implementers run without a live log but otherwise behave identically (outcome, AskUser, commit/push, PR). Session history no longer accumulates per change.
- **Dependencies:** `a56` (the `CliStrategy` trait, `agentic_run`, the `outcome_*` relay) AND `a55` (registry). Composes with `a60` (`OpencodeStrategy`) and `a69` (`GeminiStrategy`) to make those CLIs implementers; processes after them.
- **Acceptance:** `cargo test` passes; `cargo clippy --all-targets -- -D warnings` is clean; `openspec validate a70-capture-mode-implementer --strict` passes. The integration spike confirms each non-claude strategy's headless resume mechanism AND the exact session-delete that targets only the created session (leaving settings/memory/auth intact).
