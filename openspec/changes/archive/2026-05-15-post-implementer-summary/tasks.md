## 3. polling_loop: build summary and post comment

- [x] 3.1 Add `fn build_implementer_summary(workspace: &Path, processed: &[String]) -> String` in `polling_loop.rs`. For each change in `processed`, read `crate::executor::claude_cli::run_log_path(workspace, change)`, extract the `=== STDOUT (n bytes) ===` section by finding that header and the next `=== STDERR (...) ===` header (or end-of-file). Strip trailing whitespace. If the file is unreadable, log WARN and skip that change. If the change's stdout is empty (after strip), set the section content to the literal `_(no implementer output captured)_`. Concatenate into the markdown shape:
    ```
    ## Agent implementation notes
    <!-- Generated by autocoder; the agent's end-of-run output for each
         change in this pass. Reviewer is the separate `## Code Review`
         section in the PR body. -->

    ### <change-1>

    <stdout-1>

    ### <change-2>

    <stdout-2>
    ```
    If ALL changes' logs are unreadable, return an empty string (caller skips the post).
- [x] 3.2 Add `fn truncate_to_fit(body: String, max: usize) -> String`. If `body.len() <= max`, return as-is. Otherwise truncate to a char-boundary at or below `max`, then append `\n\n_[implementer summary truncated to fit GitHub comment limit; full output at /tmp/autocoder/logs/<workspace-basename>/<change>.log]_`. Use `max = 60_000`.
- [x] 3.3 In `open_pull_request` (or just after its call in `execute_one_pass`), after the PR creation returns Ok with the `CreatedPr.number`, build the summary, truncate it, and call `create_issue_comment(upstream_owner, upstream_repo, pr_number, body, token)`. Wrap the call in `if let Err(e) = ... { tracing::error!("posting implementer-summary comment failed: {e:#}") }`. The PR is already created — never propagate this error.
- [x] 3.4 Skip the comment when `build_implementer_summary` returns an empty string (all-logs-missing case). Log INFO naming the skip.
