## Why

When the implementer agent finishes a change, its final stdout often contains a useful summary: modules added, test counts, deviations from the spec it had to make, and sometimes meta-observations (e.g. "I noticed a pre-existing bug in path X"). Today that intel lives only in `/tmp/autocoder/logs/<basename>/<change>.log` and is invisible to anyone reviewing the PR. A reviewer asking "what changed and why?" has the diff and the code-reviewer's verdict; they do NOT have the implementer's own narrative.

Posting that narrative as a PR comment puts the implementer's voice directly on the PR page where reviewers can read it (or collapse it) without SSHing to the daemon host.

## What Changes

- **MODIFIED capability:** `git-workflow-manager` — after `open_pull_request` succeeds, autocoder SHALL post a single follow-up issue comment to the new PR containing one section per archived change, each carrying the captured stdout from that change's implementation run. The comment is best-effort: a failure to post does NOT fail the PR creation.
- **New GitHub call:** `github::create_issue_comment(api_base, owner, repo, issue_number, body, token)` posting to `POST /repos/{owner}/{repo}/issues/{issue_number}/comments`. Returns Ok on 2xx, Err on non-2xx (with status code + 500-char body snippet, matching existing helpers).
- **Source of the summary:** for each archived change in the pass, read `<system-temp>/autocoder/logs/<workspace-basename>/<change>.log`, extract the `=== STDOUT (n bytes) ===` block (only stdout — stderr is operator-facing log noise, not implementer narrative). Skip changes whose log file is missing or unreadable; log WARN, continue.
- **Comment body format:**
    ```markdown
    ## Agent implementation notes
    <!-- Generated by autocoder; the agent's end-of-run output for each
         change in this pass. Reviewer is the separate `## Code Review`
         section in the PR body. -->

    ### chatops-progress-notifications

    <stdout for that change>

    ### experimental-chatops-providers

    <stdout for that change>
    ```
- **Size cap:** total comment body is truncated at 60,000 characters (GitHub's hard limit is 65,535; we keep headroom for our wrapper text and markdown formatting). When truncated, the truncated tail is replaced with `\n\n_[implementer summary truncated to fit GitHub comment limit; full output at /tmp/autocoder/logs/<basename>/<change>.log]_`.
- **PR number plumbing:** `github::create_pull_request` currently returns the PR's `html_url` only. Extend it to return a struct `{ html_url, number }` so callers have the PR number for the follow-up comment call. Update all current callers and tests.
- **Empty stdout handling:** if the captured stdout is empty (the agent produced changes silently — rare but possible), the change's section in the comment is `_(no implementer output captured)_`.
- **Failure isolation:** the comment posting wraps in a `match` that logs ERROR on failure and returns `Ok(())` from `execute_one_pass` regardless. The PR's existence is the contract; the comment is enrichment.

## Impact

- Affected specs: `git-workflow-manager` (PR creation requirement adjusted to also return PR number; new requirement for follow-up comment).
- Affected code: `autocoder/src/github.rs` (new helper, return-type change), `autocoder/src/polling_loop.rs` (call site after `open_pull_request`).
- One extra HTTP call per polling pass that produced commits. Negligible.
- Visible to reviewers: a single PR comment from autocoder's PAT identity, posted immediately after PR open. The comment can be collapsed or deleted by anyone with appropriate repo permissions; it does not block merge.
- Breaking? Internal callers of `github::create_pull_request` update to the new return type — bounded surface, all in this codebase.
- The implementer's narrative on the PR page also closes the visibility gap for changes where the test plan or deviation rationale would otherwise live only in a server-local log file.
