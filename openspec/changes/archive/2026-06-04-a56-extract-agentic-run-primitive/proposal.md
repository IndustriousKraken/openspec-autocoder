## Why

The "wrap a CLI as a subprocess, hand it a prompt, let it run an agentic session" pattern is implemented five times: the executor's `run_subprocess` (rich — JSON streaming, `final_answer`, session-id, MCP) plus four near-identical audit copies (drift, architecture-consultative, documentation, specs-writing), each redeclaring `SubprocessOutcome` and its own timeout/spawn/capture logic. The reviewer and contradiction-check are about to become the sixth and seventh. Rule-of-three is long past; the duplication is where divergent bugs hide (e.g. the porcelain-parse class, the streaming-vs-capture split).

This change extracts the one primitive every agentic role will share, abstracts CLI selection behind a `CliStrategy` trait (so a model's provider can pick `claude` today and `opencode` later without touching role code), and lays down the structured-submission MCP infrastructure the agentic roles use to return validated results instead of stdout-JSON. It is the architectural spine: changes 4–8 hang off it. It is behavior-neutral — the executor and audits run exactly as before; the new submission surface is additive and unused until a role adopts it.

## What Changes

**A shared agentic-run primitive (executor).** One `agentic_run` entry point takes: the workspace, a `CliStrategy`, the prompt, the sandbox (allowed-tools + disallowed patterns), an optional MCP configuration (which tools to expose + the control-socket relay env), an output mode (streaming-JSON vs simple-capture), and a timeout. It builds the invocation through the strategy, spawns in its own process group, pipes the prompt on stdin, captures stdout/stderr, enforces the timeout, and returns a unified `AgenticRunOutcome { timed_out, exit_status, stdout, stderr, final_answer: Option, session_id: Option, streamed_log }`. The executor's `run_subprocess` and the four audit copies are refactored onto it with NO behavior change: the executor keeps streaming-JSON + MCP + recovery; the audits keep simple-capture + no-MCP + their read-only tool lists.

**A `CliStrategy` trait with the claude implementation (executor).** The strategy does two jobs: (1) build the invocation — binary, flags, the `--allowedTools` / sandbox-settings format, and the MCP-config-file format; (2) translate a resolved `(provider, model, api_base_url, api_key)` into that CLI's model-selection mechanism. The `claude` strategy reproduces today's invocation exactly (`--settings`, `--allowedTools`, `--permission-mode acceptEdits`, optional `--verbose --output-format stream-json`, `.mcp.json`) and selects the model via `ANTHROPIC_BASE_URL` / `ANTHROPIC_AUTH_TOKEN` / `ANTHROPIC_MODEL` — but only when a model is configured; with no model it passes nothing, preserving the executor's current "CLI default model" behavior. The strategy for a role is resolved from the model's provider via a55's `provider → default CLI` rule.

**Per-role structured-submission MCP infrastructure (executor + orchestrator-cli).** The MCP server gains the framework for a per-role `submit_*` tool that relays a schema-validated payload to the daemon over the control socket — mirroring the existing `outcome_*` / `record_outcome` pattern. An `ORCH_MCP_ROLE` env (written into `.mcp.json`) tells the MCP child which role it serves, so it advertises only that role's submission tool. The control socket gains `record_submission` / `consume_submission` actions for execution-scoped submission storage. This change establishes the framework and the relay; the concrete per-role tools (`submit_findings`, `submit_review`, `submit_contradictions`, `submit_verdict`) and their schemas are added by the changes that consume them (4/5/6/8), each following this pattern.

## Impact

- **Affected specs:**
  - `executor` — ADDED `Shared agentic-run primitive`, `CliStrategy trait with the claude implementation`, AND `Per-execution MCP child exposes a per-role submission tool via control-socket relay`.
  - `orchestrator-cli` — ADDED `Control socket exposes record_submission AND consume_submission actions`.
- **Affected code:**
  - New module (e.g. `autocoder/src/agentic_run.rs`) with `agentic_run`, `AgenticRunOutcome`, the `CliStrategy` trait, and `ClaudeStrategy`.
  - `autocoder/src/executor/claude_cli.rs` — `run_subprocess` becomes a thin caller of `agentic_run` with `ClaudeStrategy` + streaming + MCP; the duplicated `SubprocessOutcome` is replaced by `AgenticRunOutcome`.
  - `autocoder/src/audits/{drift,architecture_consultative,documentation_audit,specs_writing}.rs` — each local `run_subprocess` + `SubprocessOutcome` deleted; callers use `agentic_run` with `ClaudeStrategy` + simple-capture + no-MCP + their existing tool lists. `audits/mod.rs::write_sandbox_settings` (already shared) is reused by the primitive.
  - `autocoder/src/mcp_askuser_server.rs` — add the `submit_*` advertisement framework keyed on `ORCH_MCP_ROLE`, and a `relay_submission` helper paralleling `relay_record_outcome`.
  - `autocoder/src/control_socket.rs` — `record_submission` / `consume_submission` actions.
- **Operator-visible behavior:** none. The executor and audits behave identically; the submission surface is unused until a role adopts it.
- **Acceptance:** `cargo test` passes; `openspec validate a56-extract-agentic-run-primitive --strict` passes. Tests: the executor path through `agentic_run` produces the same streaming log + outcome as before; an audit path produces the same simple-capture outcome; `ClaudeStrategy` with no model passes no `ANTHROPIC_MODEL` (CLI default) and with a model sets the three env vars; a `submit_*` call relays a payload to `record_submission` and `consume_submission` returns it.
- **Dependencies:** loosely on **a55** (the `provider → default CLI` rule the strategy resolution reads). Independent of a47–a53. The spine for changes 4–8.
