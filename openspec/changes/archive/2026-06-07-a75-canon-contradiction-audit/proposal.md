## Why

Two pre-flight gates already keep *new* work from contradicting the project: the `[in]` gate (`a19`/`a59`) catches a change that contradicts itself, and the `[canon]` gate (`a62`) catches a change that contradicts existing canon. Neither can catch a contradiction that already lives *inside* the canon ‚Äî two canonical requirements that cannot both hold, landed before the gates existed or slipped past them (the classic "all data in a relational database" in one requirement, "store the records in MongoDB" in another). Those sit silently until an implementer is handed mutually-impossible instructions.

This change adds the missing sibling: a periodic audit that scans the whole canon for internal contradictions and reports them advisorily, so a maintainer can heal them on purpose. It completes the contradiction family ‚Äî `within a change` (a59), `change vs canon` (a62), `canon vs canon` (here).

The canon is large (this project alone is ~12,600 lines of spec), so an all-pairs sweep is intractable and a single read-everything pass satisfices. The audit leans on `a21`'s canonical-spec RAG ‚Äî which already vectorizes the canon ‚Äî to retrieve, per requirement, the handful of semantically-nearest requirements and check that focused bundle, because contradictions live between *related* requirements (both about storage, both about auth), not random pairs.

## What Changes

**`canon_contradiction_audit` (orchestrator-cli).** A new registered periodic audit. It runs the wrapped agent CLI through the shared `agentic_run` primitive (`a56`) in a read-only sandbox (`Read`/`Glob`/`Grep`, NO `Bash`/`Write`/`Edit`) with `ORCH_MCP_ROLE = canon_contradiction_audit`, enumerates the canonical requirements, and ‚Äî when `a21`'s RAG is enabled ‚Äî retrieves the nearest requirements per requirement via `query_canonical_specs` and checks each focused bundle for pairs that cannot both hold. When RAG is not configured it degrades to a best-effort direct read of the canon and logs that coverage is best-effort. It is `requires_head_change = true` and `WritePolicy::None`, with a heavy default cadence (`monthly`) given its whole-canon scope.

**Advisory, never auto-healing.** The audit cannot know which side of a contradiction reflects project intent (is the relational requirement the law and Mongo the mistake, or has the project moved to Mongo and the relational requirement gone stale?). So it returns `AuditOutcome::Reported`; each finding names BOTH requirements (capability + title) and explains the conflict. Healing flows through the path the framework already has ‚Äî the maintainer reviews the chatops thread and, when they want it fixed, uses the existing audit-thread `send it` to schedule a triage run that drafts the resolving MODIFY in the direction they choose. The audit itself never writes the canon.

**Precision over recall.** A finding requires that the two requirements be logically incompatible. A general rule plus a *compatible* specialization (e.g. "all data in a relational database" + "use PostgreSQL", since PostgreSQL is relational) is NOT a contradiction and is not reported ‚Äî that is exactly the general-vs-specific information-loss trap, and flagging it would be noise. The prompt confidence-gates toward not reporting when uncertain.

**Re-report suppression.** An unhealed contradiction must not re-spam chatops every run. The audit persists the pairs it has reported (keyed by an order-independent capability+requirement-title pair, plus a content hash of each requirement) and suppresses a pair already recorded whose requirements are textually unchanged; it re-surfaces a pair when either requirement's text changed since it was recorded, and prunes records for pairs no longer detected.

**`submit_canon_internal_contradictions` MCP tool (executor).** Built on `a56`'s per-role submission framework, advertised only for the `canon_contradiction_audit` role. Payload `{ contradictions: [{ capability_a, requirement_a, capability_b, requirement_b, summary }] }` ‚Äî symmetric (both sides canonical), distinct from `a62`'s `submit_canon_contradictions`, which names a change requirement against a canonical one. A missing submission is consumed as empty (clean canon).

## Impact

- **Affected specs:**
  - `orchestrator-cli` ‚Äî ADDED `Canon-internal contradiction audit`; MODIFIED `Registered periodic audits` (adds `canon_contradiction_audit` to the canonical enumeration, per that requirement's own "adding an audit MUST update this requirement" mandate).
  - `executor` ‚Äî ADDED `submit_canon_internal_contradictions MCP tool returns canon-internal contradictions`.
- **Affected code:** a new audit in the `AuditRegistry` reusing the agentic-audit machinery (`agentic_run`, read-only sandbox, `submit_*` relay); RAG-assisted detection driver (enumerate requirements ‚Üí `query_canonical_specs` neighbors ‚Üí focused pairwise check), with a logged best-effort fallback when RAG is off; report-state persistence for suppression in `.audit-state.json`; `submit_canon_internal_contradictions` registered in the MCP server gated on the role; embedded prompt `prompts/canon-contradiction-audit.md`; `validate_audit_type_names` known-slug list, the README audit table, and `config.example.yaml` updated to keep spec/validator/docs aligned (per the registry requirement's mandate).
- **Operator-visible behavior:** none unless enabled (defaults `disabled` like every audit). When enabled, a canon-internal contradiction surfaces as a chatops finding naming both requirements; the maintainer heals it via `send it` in the thread. Wizard fast-path integration is intentionally out of scope for this change ‚Äî the audit is enabled via config like any other.
- **Dependencies:** uses `a21` (`query_canonical_specs`) opportunistically (strongly recommended, degrades without it), `a56` (`agentic_run` + the `submit_*` framework), the periodic-audit framework, and the existing audit-thread `send it` triage path. Structurally parallels `a59`/`a62`. No stacking ‚Äî all dependencies are canonical.
- **Acceptance:** `cargo test` passes; `cargo clippy --all-targets -- -D warnings` is clean; `openspec validate a75-canon-contradiction-audit --strict` passes. Tests: the audit registers and runs read-only/agentic with the role; it uses `query_canonical_specs` when RAG is on and logs best-effort when off; a general+compatible-specific pair is NOT reported; a genuinely incompatible pair is reported advisorily naming both requirements with the canon unmodified; a previously-reported unchanged pair is suppressed while an edited one re-surfaces and a healed one is pruned; an empty result is silent; `submit_canon_internal_contradictions` is advertised only for the role and a missing submission consumes as empty.
